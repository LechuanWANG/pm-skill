# Prompt 和 Context 设计

## 目标

用尽量短的 prompt、清晰的结构化输入和明确的输出契约，让模型完成具体任务。不要把稳定业务规则、校验逻辑和权限边界全部塞进 prompt。

## 设计顺序

1.  判断任务是否真的需要模型；能用确定性代码解决的部分，不要交给模型自由判断。
2.  定义输入：哪些信息必须进入上下文，哪些可以放在外部系统、检索、数据库或文件中。
3.  定义输出：优先使用 schema、字段约束、枚举值、结构化 JSON 或表格。
4.  定义上下文预算：长文档、聊天历史、上传文件、多步骤 Agent 都要限制进入模型的信息。
5.  定义校验：哪些输出必须由确定性代码、schema、测试、人工确认或业务规则检查。
6.  定义失败处理：输出不合规、信息不足、检索失败、模型不确定时如何 fallback。

## Prompt 规则

-   system prompt 只放稳定角色、边界和不可违反的行为约束。
-   user prompt 放本次任务、输入数据、目标和输出要求。
-   不要在 system、developer、user prompt 之间重复堆同一条规则，除非运行时确实需要。
-   few-shot 示例只有在明显改善行为时才加入。
-   prompt 越写越长时，优先考虑把规则移到 schema、validator、tool contract、retrieval 或确定性代码中。
-   避免把所有资料硬塞进 prompt，优先使用按需读取、总结、检索和引用。
-   尽量少用否定句 prompt，多写正面的引导性 prompt

## Context 策略

-   对长文档，先建立索引或摘要，再按任务读取相关部分。
-   对聊天历史，保留决策、约束、用户纠正和当前任务必要背景，丢弃重复寒暄和过期推测。
-   对文件和数据，优先传结构化片段、路径、schema 或查询结果，不传无关全文。
-   对多步骤 Agent，把阶段性状态压缩成任务需要的事实、未解决问题和下一步动作。
-   对项目内 LLM Wiki，先读 `docs/ai-wiki/index.md`，再读相关页面，不默认加载整个知识库。

## 输出契约

当 AI 输出会进入产品流程、数据库、工具调用或后续自动化时，必须定义输出契约：

-   字段名、类型、必填项和枚举值。
-   不允许输出的内容。
-   缺信息时的返回格式。
-   置信度、证据、引用或解释字段是否需要。
-   下游代码如何验证和处理失败。

## 避免

-   不要用长篇自然语言替代 schema。
-   不要把 prompt 当作权限系统。
-   不要让模型在没有数据来源和校验机制时编造事实。
-   不要为了“智能”让模型自由决定本应由产品规则决定的事情。
