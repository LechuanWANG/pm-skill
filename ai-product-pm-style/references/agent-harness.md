# Agent 和 Harness 设计

## 目标

通过 Harness 工程将 Agent 设计成灵活、上下文合理、有技术支撑的智能体，服务于产品。

## 是否需要 Agent

先判断任务是否真的需要 Agent：

-   如果流程固定、输入清晰、输出确定，优先使用普通服务、函数、队列或工作流。
-   如果任务需要多步判断、动态选择工具、处理不完整信息或根据中间结果调整路径，才考虑 Agent。
-   如果 Agent 只会增加不可控性、延迟或维护成本，不要为了形式引入。

## Agent/tool 策略

-   先从用户流程和产品价值判断 Agent 的职责，不要从框架能力反推产品。
-   LangChain、LangGraph 或类似 orchestration 架构在带来明确质量、效率、可扩展性或可维护性收益时要使用。
-   工具设计要小而清晰，参数结构化，错误返回可恢复。
-   对写操作、外部副作用、付费操作、删除操作和发布操作，默认要求确认或强校验。
-   对长任务，拆成可观察阶段，保留中间状态和恢复路径。

## 开源项目和外部研究

当开源项目、prototype 或公开实现会明显影响产品质量、架构选择、实现效率或风险判断时，主动搜索 GitHub 和外部资料。

搜索时关注：

-   Agent 编排方式。
-   Harness 约束。
-   tool contract。
-   evaluation 方法。
-   数据处理链路。
-   UI 交互模式。
-   工程目录组织。

借鉴开源项目时，优先吸收产品思路、架构模式、evaluation 方法、UX 流程或工程组织方式。直接引入依赖或复用代码片段前，必须确认许可证、来源链接、维护状态、依赖兼容性、安全风险和项目适配成本。需要记录借鉴思路时，使用 `references/open-source-ideas.md` 的格式。

## 评估方式

Agent 方案必须说明如何检查：

-   任务成功率和输出质量。
-   延迟、成本和失败率。
-   工具调用是否符合权限和顺序。
-   fallback 是否生效。
-   用户是否能理解、编辑、接受、重试或纠正 AI 输出。
-   出错时是否有足够证据定位问题。
