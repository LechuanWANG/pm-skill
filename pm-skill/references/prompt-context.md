# Prompt 和 Context 设计

## 目标

用短 prompt、结构化输入和明确输出契约完成任务。稳定业务规则、权限边界和校验逻辑优先放入 schema、validator、tool contract、retrieval 或确定性代码。

## 设计顺序

1.  判断模型职责：推理、起草、分类、总结、规划、检索增强或工具选择。
2.  定义输入：必须进入上下文的信息、可按需检索的信息、适合保留在外部系统的信息。
3.  定义输出：字段名、类型、必填项、枚举值、缺信息格式和证据要求。
4.  定义上下文预算：长文档、聊天历史、上传文件、多步骤 Agent 都要裁剪到任务相关内容。
5.  定义校验：schema、测试、业务规则、人工确认或确定性代码各负责什么。
6.  定义失败处理：信息缺失、检索失败、输出违规、模型低置信时的 fallback。

## Prompt 规则

-   system prompt 放稳定角色、边界和固定行为约束。
-   user prompt 放本次任务、输入数据、目标和输出要求。
-   few-shot 示例只在明显改善输出时加入。
-   prompt 变长时，把规则迁移到 schema、validator、tool contract、retrieval 或代码。
-   大资料先索引、摘要、检索和引用，再进入模型上下文。
-   Prompt 尽量用正向引导句，减少否定句堆叠。

## Context 策略

-   长文档：先建索引或摘要，再按任务读取相关部分。
-   聊天历史：保留决策、约束、用户纠正和当前任务背景。
-   文件和数据：传结构化片段、路径、schema 或查询结果。
-   多步骤 Agent：把阶段性状态压缩成事实、未解决问题和下一步动作。
-   LLM Wiki：先读 `docs/llm-wiki/index.md`，再读相关页面和必要 raw source。

## 输出契约

当 AI 输出进入产品流程、数据库、工具调用或自动化时，定义：

-   字段名、类型、必填项和枚举值。
-   受限内容和缺信息返回格式。
-   置信度、证据、引用或解释字段。
-   下游验证和失败处理。

## 设计检查

-   Schema 承担格式约束。
-   Validator 承担合规检查。
-   Tool contract 承担权限和参数边界。
-   Retrieval 承担资料选择。
-   产品规则承担业务判断。
