# Brainstorming 工作方式

## 目标

做 AI 产品 brainstorming 时，先帮助用户判断产品价值、用户流程、AI 职责、约束条件、技术取舍和可落地路径。

## 推荐流程

1.  明确用户和场景：谁在用，要完成什么任务，当前成本或质量问题是什么。
2.  定义产品结果：用户最终看到什么、得到什么、能节省什么或提升什么。
3.  划分 AI 职责：AI 负责判断、起草、分类、检索、总结、规划还是执行。
4.  判断是否需要 Agent：只有多步判断、动态工具选择、信息缺口或中间结果会改变路径时才引入 Agent；固定流程优先用普通服务、函数、队列或工作流。
5.  标出 Harness 边界：AI 负责什么，确定性代码负责什么，哪些决策交给用户确认。
6.  设计数据和上下文：哪些信息进入模型，哪些留在外部，哪些按需 retrieval 或总结。
7.  搜索可借鉴的 GitHub prototype 或开源实现：在技术路线、交互模式、Agent/RAG/Harness 或产品形态不确定时，主动找相近项目提炼做法。
8.  选择语言和技术栈：根据软件特性、公开实现启发和执行计划推荐计算机语言、框架、部署路径和备选方案。
9.  梳理产品界面：用户如何查看、编辑、接受、重试或纠正 AI 输出。
10. 拆出 MVP 和里程碑：先验证最小闭环，再扩展复杂能力。
11. 定义评估方式：质量、延迟、成本、可用性、失败场景和用户反馈如何检查。

## AI 职责优先

先定义用户流程中的 AI 职责，再选择 Agent、workflow、RAG 或普通 API。不要从框架出发倒推产品。

判断顺序：

-   AI 的用户可见价值是什么：判断、生成、解释、检索、规划、执行还是陪伴。
-   这个职责是否需要多步状态、工具选择、动态分支或长期上下文。
-   确定性代码、规则、schema、检索、数据库和人工确认分别承担什么。
-   Agent 带来的质量收益是否值得额外的延迟、成本、调试和可解释性成本。
-   多 Agent 只用于明显需要多视角、交叉审查、协作规划或复杂综合的高价值环节。

## GitHub prototype 默认触发

Brainstorming 新 AI 产品、重大功能、Agent/RAG/workflow、产品 UI 模式或技术栈时，默认考虑 GitHub prototype 搜索。满足任一条件就主动搜索，除非用户明确不要联网、任务明显只是离线小改，或项目隐私/合规边界不允许：

-   用户在探索新方向、MVP、竞品形态、技术路线或可行性。
-   方案依赖 Agent orchestration、tool calling、RAG、实时流式输出、文件处理、数据可视化、复杂 UI 或部署架构。
-   你准备推荐某个框架、语言、服务、目录结构或交互模式，但缺少现实实现参照。
-   公开 prototype 可能显著影响产品方案、工程组织、评估方式、交互细节、成本或风险判断。
-   用户提到“类似”“竞品”“prototype”“开源”“GitHub”“怎么做”“最佳实践”或给出一个产品方向但没有实现参照。

搜索后不要堆链接。筛 2-5 个最相关项目，提炼产品启发、架构模式、可复用交互、evaluation 方法、适配成本和 license/安全风险。记录方式见 `references/open-source-ideas.md`；有长期价值时同步写入项目 `docs/llm-wiki/`。

## 语言和技术栈选择

需要制作 app、网站、工具、SaaS 或原型时，要评估计算机语言与技术栈。

评估维度：

-   目标平台：Web、移动端、桌面端、CLI、浏览器扩展、后端服务、数据管道。
-   交互复杂度：表单、编辑器、实时协作、仪表盘、聊天界面、可视化、工作流编排。
-   AI/API 集成：模型 API、streaming、tool calling、RAG、异步任务、队列、回调。
-   数据处理：结构化数据、文件解析、向量检索、批处理、实时分析、权限隔离。
-   实时性和性能：低延迟 UI、长任务、并发、后台任务、GPU/本地模型需求。
-   部署方式：Vercel、Cloudflare、容器、私有化部署、serverless、长期后台 worker。
-   团队迭代速度：现有代码库、团队熟悉度、生态成熟度、调试效率。
-   长期维护成本：类型安全、测试、依赖稳定性、社区活跃度、招聘和迁移成本。

输出技术选型时，给出：

-   推荐语言/框架。
-   选择理由和放弃理由。
-   该方案对产品体验、成本、延迟、维护和未来扩展的影响。

## 重大更新

当任务影响核心产品方向、系统架构、AI 行为、数据模型、prompt 策略、UI 主流程，或者用户上传较大的需求文档、项目说明或资料文件时，把它视为重大更新。

重大更新时：

-   先认真阅读用户提供的上下文。
-   产品目标、目标用户、核心约束、数据来源、AI 行为、验收标准或边界条件存在缺口时，追问关键问题。
-   对主路径影响较小的问题，明确假设后继续推进。
-   在理解充分后，给出可执行阶段计划和可验证里程碑。
