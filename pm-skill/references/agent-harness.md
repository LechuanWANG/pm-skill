# Agent 和 Harness 设计

## 目标

在设计新的 Agent 时，用 Harness 工程把 Agent 设计成可控、可观察、可恢复的产品能力。

## Agent 适用性

-   固定流程、清晰输入、确定输出：优先使用普通服务、函数、队列或工作流。
-   多步判断、动态工具选择、信息缺口、中间结果会改变路径：适合 Agent。
-   延迟、成本、可解释性和维护成本高于收益时，选择更简单的确定性方案。

## Agent/tool 策略

-   从用户流程和产品价值定义 Agent 职责。
-   LangChain、LangGraph 或类似 orchestration 架构用于提升质量、效率、可扩展性或可维护性。
-   工具保持清晰：参数结构化，错误可恢复，返回值可验证。
-   写操作、外部副作用、付费、删除和发布操作使用确认或强校验。
-   长任务拆成可观察阶段，保留中间状态和恢复路径。

## Harness 检查表

设计 Agent Harness 时明确：

-   状态：当前任务阶段、历史动作、用户输入、可用证据和剩余预算。
-   输入：哪些字段必须结构化，哪些上下文按需检索，哪些数据不能进入模型。
-   输出：schema、枚举、证据要求、缺信息格式和下游消费方。
-   工具：参数边界、权限、幂等性、失败返回和可恢复策略。
-   阶段转换：什么条件继续、追问、切换、等待用户、结束或回滚。
-   校验：schema、业务规则、重复检测、权限检查、人工确认和测试各负责什么。
-   观察：日志、事件、阶段进度、成本、延迟和出错时可定位的证据。
-   持久化：哪些中间状态必须保存，哪些只是临时上下文，哪些需要长期记忆。

## Agent 执行模式

复杂 AI 行为优先拆成小节点，而不是交给一个长 prompt 一次完成：

```text
analyze -> decide -> retrieve -> generate -> validate -> persist
```

-   `analyze`：读取当前状态，提取事实、限制、用户意图和风险。
-   `decide`：选择下一步动作、技能、工具、阶段或是否等待用户。
-   `retrieve`：只有需要证据或长期上下文时才检索，返回小而可引用的上下文包。
-   `generate`：生成用户可见内容或工具参数。
-   `validate`：检查 schema、重复、越权、证据缺失、格式和业务规则。
-   `persist`：保存流程必需状态、用户确认、关键证据和长期有价值的结论。

能用确定性代码完成的节点不要放进模型。模型主要负责判断、综合、表达和难以规则化的选择。

## 开源项目和外部研究

当公开实现会影响产品质量、架构选择、实现效率或风险判断时，主动研究 GitHub 和外部资料。

重点关注：

-   Agent 编排方式。
-   Harness 约束。
-   Tool contract。
-   Evaluation 方法。
-   数据处理链路。
-   UI 交互模式。
-   工程目录组织。

借鉴开源项目时，吸收产品思路、架构模式、evaluation 方法、UX 流程和工程组织方式。直接引入依赖或复用代码片段前，确认许可证、来源链接、维护状态、依赖兼容性、安全风险和适配成本。记录格式见 `references/open-source-ideas.md`。

## 评估方式

Agent 方案说明：

-   任务成功率和输出质量。
-   延迟、成本和失败率。
-   工具调用权限、参数和顺序。
-   fallback 和恢复路径。
-   用户查看、编辑、接受、重试、纠正 AI 输出的入口。
-   出错时可定位的证据。
