# Agent 和 Harness 设计

## 目标

用 Harness 工程把 Agent 设计成可控、可观察、可恢复的产品能力。

## Agent 适用性

-   固定流程、清晰输入、确定输出：优先使用普通服务、函数、队列或工作流。
-   多步判断、动态工具选择、信息缺口、中间结果会改变路径：适合 Agent。
-   延迟、成本、可解释性和维护成本高于收益时，选择更简单的确定性方案。

## Agent/tool 策略

-   从用户流程和产品价值定义 Agent 职责。
-   LangChain、LangGraph 或类似 orchestration 架构用于提升质量、效率、可扩展性或可维护性。
-   工具保持小而清晰：参数结构化，错误可恢复，返回值可验证。
-   写操作、外部副作用、付费、删除和发布操作使用确认或强校验。
-   长任务拆成可观察阶段，保留中间状态和恢复路径。

## 开源项目和外部研究

当公开实现会影响产品质量、架构选择、实现效率或风险判断时，主动研究 GitHub 和外部资料。

重点关注：

-   Agent 编排方式。
-   Harness 约束。
-   Tool contract。
-   Evaluation 方法。
-   数据处理链路。
-   UI 交互模式。
-   工程目录组织。

借鉴开源项目时，吸收产品思路、架构模式、evaluation 方法、UX 流程和工程组织方式。直接引入依赖或复用代码片段前，确认许可证、来源链接、维护状态、依赖兼容性、安全风险和适配成本。记录格式见 `references/open-source-ideas.md`。

## 评估方式

Agent 方案说明：

-   任务成功率和输出质量。
-   延迟、成本和失败率。
-   工具调用权限、参数和顺序。
-   fallback 和恢复路径。
-   用户查看、编辑、接受、重试、纠正 AI 输出的入口。
-   出错时可定位的证据。
